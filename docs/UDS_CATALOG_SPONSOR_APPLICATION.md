## 4. SLSA L3 Transition Plan

### Current State: SLSA L1 Honest
We acknowledge our current attestation is L1: we generate provenance but it is self-attested (not non-forgeable). This is documented honestly in our release notes.

### Target: SLSA L3 by v0.4.0-rc.1

**Steps:**

**Step 1 — slsa-github-generator Adoption (v0.4.0-alpha.1)**
```yaml
# .github/workflows/uds-package-release.yml addition
jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      hashes: ${{ steps.hash.outputs.hashes }}
    steps:
      - name: Build Zarf package
        run: make zarf-package-create
      - name: Generate subject hashes
        id: hash
        run: |
          sha256sum *.zst | base64 -w0 > hashes.txt
          echo "hashes=$(cat hashes.txt)" >> $GITHUB_OUTPUT

  provenance:
    needs: [build]
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@v2
    with:
      base64-subjects: "${{ needs.build.outputs.hashes }}"
    secrets: inherit
    permissions:
      id-token: write   # required for OIDC → Fulcio
      contents: write
      actions: read
```

**Step 2 — Sigstore Fulcio Keyless (already operational via Zarf v0.77.0)**
Zarf's keyless signing uses Fulcio OIDC-authenticated ephemeral certificates. The Rekor transparency log entry is created automatically. No additional Sigstore config needed beyond current `uds-package-release.yml`.

**Step 3 — Verification Command for DU Reviewers**
```bash
# Verify SLSA L3 provenance (add to VERIFY.md)
slsa-verifier verify-artifact \
  uds-package-szl-organs-v0.4.0.tar.zst \
  --provenance-path uds-package-szl-organs-v0.4.0.tar.zst.intoto.jsonl \
  --source-uri github.com/szl-holdings/szl-uds-deployment \
  --source-tag v0.4.0
```

**Step 4 — Rekor Entry Pinning**
Each release will publish its Rekor log index in `RELEASE_NOTES.md` so verifiers can independently confirm the transparency log entry without trusting SZL.

**Step 5 — SLSA Gate in CI**
A post-build verification job will run `slsa-verifier` against its own artifact before tagging. If verification fails, the tag is not pushed. This prevents accidental L1-only releases.

## 7. Air-Gap Testing Methodology

### v0.3.1 Method (manual)
Air-gap fidelity demonstrated in `demo/warhacker_demo.sh` — network policy applied manually, egress visually confirmed. Adequate for demo; insufficient for CI.

### v0.4.0 Method (CI-enforced)

**Tool stack:**
- k3d (local Kubernetes in Docker)
- `kubectl apply -f manifests/deny-all-egress.yaml` (NetworkPolicy + Calico)
- `tcpdump` packet capture on k3d node: verify zero external packets during deploy
- Zarf's built-in air-gap mode (`zarf package deploy --no-progress` with pre-pulled images)

**CI job (`make airgap-test`):**
```bash
#!/usr/bin/env bash
set -euo pipefail

# 1. Start k3d cluster with no external network
k3d cluster create szl-airgap \
  --network szl-airgap-net \
  --agents 1

# 2. Apply deny-all NetworkPolicy
kubectl apply -f manifests/deny-all-egress.yaml

# 3. Pre-load zarf package (simulates sneaker-net)
zarf tools registry push \
  --plain-http \
  localhost:5000 \
  uds-package-szl-organs-v0.4.0.tar.zst

# 4. Deploy — any external call fails with NetworkPolicy
zarf package deploy uds-package-szl-organs-v0.4.0.tar.zst \
  --confirm \
  --no-progress

# 5. Verify organ datasets loaded
kubectl get configmap -n szl-organs --no-headers | wc -l | grep -q "^6$"

# 6. Publish result artifact
echo "AIRGAP_TEST=PASS" >> $GITHUB_OUTPUT
```

**Evidence artifact:** GitHub Actions job summary with `tcpdump` packet count (must be 0 external).
